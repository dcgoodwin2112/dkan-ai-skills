Scaffold a new AI Provider plugin for the Drupal AI module — adds a new LLM backend (chat, embeddings, etc.). Two flavors: extending `OpenAiBasedProviderClientBase` for OpenAI-compatible APIs (the common case, almost no code) or extending `AiProviderClientBase` directly for fully custom APIs (more code).

Read [SKILL.md](../skills/drupal-ai-module/SKILL.md), [plugin-types.md](../skills/drupal-ai-module/reference/plugin-types.md#ai-provider), [pitfalls.md](../skills/drupal-ai-module/reference/pitfalls.md), and [services.md](../skills/drupal-ai-module/reference/services.md#provider-plugin) before proceeding. Then read `<webroot>/modules/contrib/ai_provider_anthropic/src/Plugin/AiProvider/AnthropicProvider.php` (OpenAI-compatible) and the Anthropic `definitions/api_defaults.yml` as references.

## Input

`$ARGUMENTS` should be: `<module_path> <ProviderName> [base] [operation_types...]`

- `module_path`: Path to the target module — relative to the project root (e.g. `<webroot>/modules/custom/my_module`) or just the machine name (e.g. `dkan_query_tools`), resolved against the site's module directories.
- `ProviderName`: PascalCase class name without the `Provider` suffix (e.g., `Mistral` → generates `MistralProvider`).
- `base`: Optional. `openai` (default — extends `OpenAiBasedProviderClientBase`) or `custom` (extends `AiProviderClientBase`). Pick `openai` if the upstream API is OpenAI-compatible.
- `operation_types`: Optional. One or more of `chat`, `embeddings`, `text_to_image`, `text_to_speech`, `speech_to_text`, `moderation`, `image_classification`, `audio_to_audio`. Default: `chat`. **For `openai` base, the base already implements all six relevant interfaces** — listing extra ops here only changes what `getSupportedOperationTypes()` returns.

## Steps

### 1. Version gate

Run `composer show drupal/ai`. Refuse on `2.x` ("breaking provider lifecycle changes; pin to ^1.3 first"). Refuse if not installed.

### 2. Locate module and derive identifiers

- Read `<module_path>/<module_name>.info.yml`.
- Provider plugin ID: snake_case of `ProviderName` (e.g., `Mistral` → `mistral`).
- Class FQN: `Drupal\<module_name>\Plugin\AiProvider\<ProviderName>Provider`.

### 3. Update `<module_name>.info.yml` dependencies

Ensure these exist:
- `ai:ai`
- `configure: <module_name>.settings_form` at the top level so the module's "Configure" link in the modules page points to your provider settings.

Optional hardening: `key:key`. Providers store the API credential as a Key entity, but the bundled `ai_provider_anthropic` does NOT declare `key:key` in its `info.yml` (it relies on `ai`/`key` already being present). Declaring it is a defensible explicit dependency, not a requirement — add it only if you want the module to fail installation when `key` is absent.

### 4. Generate `definitions/api_defaults.yml`

**Schema is keyed by operation type — NOT by `models`.** Generate one top-level entry per requested operation. Each has `input`, `authentication`, `configuration` sub-keys.

Path: `<module_path>/definitions/api_defaults.yml`. Generate this exact shape (chat example; replicate per requested op):

```yaml
chat:
  input:
    description: 'Input provided to the model.'
    type: 'array'
    default:
      - { role: "system", content: "You are a helpful assistant." }
      - { role: "user", content: "Hello." }
    required: true
  authentication:
    description: '<ProviderName> API Key.'
    type: 'string'
    default: ''
    required: true
  configuration:
    max_tokens:
      label: 'Max Tokens'
      description: 'Maximum number of tokens in the response.'
      type: 'integer'
      default: 4096
      required: false
    temperature:
      label: 'Temperature'
      description: 'Sampling temperature 0-1.'
      type: 'float'
      default: ''
      required: false
      constraints: { min: 0, max: 1, step: 0.1 }
```

Header comment: `# Generated by /ai-scaffold-provider. Replace defaults with values from <ProviderName>'s API docs.`

### 5. Generate the Provider class

Path: `<module_path>/src/Plugin/AiProvider/<ProviderName>Provider.php`.

**Critical patterns:**
- **Do NOT define a constructor.** `AiProviderClientBase::__construct()` is `final`. To add custom services, override `create()` only.
- **Do NOT redeclare operation interfaces** (ChatInterface, etc.) on the class IF extending `OpenAiBasedProviderClientBase` — the base already implements all six.
- The `#[AiProvider]` attribute has only `id`, `label`, `deriver`. **No `description` and no `module_dependencies` field** (unlike `#[AiAgent]` and `#[FunctionCall]`).

**For `base = openai`** (default — most common case):

```php
<?php

declare(strict_types=1);

namespace Drupal\<module_name>\Plugin\AiProvider;

use Drupal\Core\StringTranslation\TranslatableMarkup;
use Drupal\ai\Attribute\AiProvider;
use Drupal\ai\Base\OpenAiBasedProviderClientBase;

/**
 * Plugin implementation of the '<plugin_id>' provider.
 */
#[AiProvider(
  id: '<plugin_id>',
  label: new TranslatableMarkup('<ProviderName>'),
)]
class <ProviderName>Provider extends OpenAiBasedProviderClientBase {

  /**
   * The base API endpoint for this provider.
   *
   * @var string
   */
  protected string $endpoint = 'https://api.<provider>.com/v1';

  /**
   * {@inheritdoc}
   */
  public function getConfiguredModels(?string $operation_type = NULL, array $capabilities = []): array {
    // Return [model_id => display_name]. For dynamic providers, fetch and
    // cache via $this->cacheBackend; for static, hard-code here.
    return [
      'placeholder-model-id' => 'TODO Model Name',
    ];
  }

  /**
   * {@inheritdoc}
   */
  public function getSupportedOperationTypes(): array {
    return [<requested operation types as comma-separated quoted strings>];
  }

  /**
   * {@inheritdoc}
   */
  public function getSetupData(): array {
    return [
      'key_config_name' => 'api_key',
      'default_models' => [
        // 'chat' => 'placeholder-model-id',
      ],
    ];
  }

  /**
   * {@inheritdoc}
   */
  public function getModelSettings(string $model_id, array $generalConfig = []): array {
    // Filter or augment per-model settings here. Return $generalConfig
    // unchanged if no per-model adjustments are needed.
    return $generalConfig;
  }

}
```

**For `base = custom`** (fully custom API):

```php
<?php

declare(strict_types=1);

namespace Drupal\<module_name>\Plugin\AiProvider;

use Drupal\Core\StringTranslation\TranslatableMarkup;
use Drupal\ai\Attribute\AiProvider;
use Drupal\ai\Base\AiProviderClientBase;
use Drupal\ai\OperationType\Chat\ChatInput;
use Drupal\ai\OperationType\Chat\ChatInterface;
use Drupal\ai\OperationType\Chat\ChatOutput;

#[AiProvider(
  id: '<plugin_id>',
  label: new TranslatableMarkup('<ProviderName>'),
)]
class <ProviderName>Provider extends AiProviderClientBase implements ChatInterface {
  // Implement one *Interface per requested operation type.

  /**
   * Resolved API key value (set by setAuthentication()).
   *
   * @var string|null
   */
  protected ?string $apiKey = NULL;

  /**
   * {@inheritdoc}
   */
  public function getConfiguredModels(?string $operation_type = NULL, array $capabilities = []): array {
    return ['placeholder-model-id' => 'TODO Model Name'];
  }

  /**
   * {@inheritdoc}
   */
  public function getSupportedOperationTypes(): array {
    return [<requested operation types>];
  }

  /**
   * {@inheritdoc}
   */
  public function isUsable(?string $operation_type = NULL, array $capabilities = []): bool {
    // Return TRUE if API key is set and provider can handle the op.
    return !empty($this->apiKey);
  }

  /**
   * {@inheritdoc}
   */
  public function setAuthentication(mixed $authentication): void {
    // $authentication is typically a Key entity ID string.
    if (is_string($authentication)) {
      $key = $this->keyRepository->getKey($authentication);
      $this->apiKey = $key ? $key->getKeyValue() : NULL;
    }
  }

  /**
   * {@inheritdoc}
   */
  public function getModelSettings(string $model_id, array $generalConfig = []): array {
    return $generalConfig;
  }

  /**
   * {@inheritdoc}
   */
  public function chat(array|string|ChatInput $input, string $model_id, array $tags = []): ChatOutput {
    // TODO: Build the HTTP request via $this->httpClient, parse, return ChatOutput.
    throw new \LogicException('Implement <ProviderName>Provider::chat() against the upstream API.');
  }

}
```

For each additional operation type beyond chat, add the matching `*Interface` to the `implements` clause and the operation method (e.g., `embeddings()`).

### 6. Generate the settings form

Path: `<module_path>/src/Form/<ProviderName>ConfigForm.php`. Standard `ConfigFormBase` subclass with:

- A select for the API Key (populated from Key entities — inject `key.repository` and call `getKeys()`)
- An optional `host` text field for custom endpoints (matches Anthropic's pattern for proxies/Ollama)
- `getEditableConfigNames()` returns `['<module_name>.settings']`

### 7. Add the routing

Path: `<module_path>/<module_name>.routing.yml` (create or append):

```yaml
<module_name>.settings_form:
  path: '/admin/config/ai/providers/<plugin_id>'
  defaults:
    _form: '\Drupal\<module_name>\Form\<ProviderName>ConfigForm'
    _title: 'Setup <ProviderName> Authentication'
  requirements:
    _permission: 'administer ai providers'
```

### 8. Add config schema

Path: `<module_path>/config/schema/<module_name>.schema.yml`. Skeleton with `<module_name>.settings:` mapping containing `api_key` (Key entity ID, type `string`) and any other fields the form exposes (`host`, etc.).

### 9. Add config defaults

Path: `<module_path>/config/install/<module_name>.settings.yml`. **Empty placeholder strings only** — no raw API keys. The user must create a Key entity at `/admin/config/system/keys` and select it in the settings form post-install.

### 10. Generate a unit test stub

Path: `<module_path>/tests/src/Unit/Plugin/AiProvider/<ProviderName>ProviderTest.php`. Test that:

- The plugin attribute exposes the expected `id` and `label`.
- `getSupportedOperationTypes()` returns the expected operations.

Use `newInstanceWithoutConstructor()` to bypass the `final` 8-service constructor — for behavior tests, write a kernel test instead.

### 11. Cache rebuild

Run `ddev drush cr`.

### 12. Verify runtime discovery

Don't report done on scaffold output alone. Provider instantiation requires a Key entity, but discovery can be checked without one:

```bash
ddev drush ev "\$d = \Drupal::service('ai.provider')->getDefinitions()['<plugin_id>'] ?? NULL; echo \$d ? \$d['class'] . PHP_EOL : 'NOT FOUND' . PHP_EOL;"
```

Expected: prints the provider class FQN. `NOT FOUND` means the attribute didn't parse, the directory casing is wrong, or `drush cr` was skipped. After creating a Key entity, also run `\Drupal::service('ai.provider')->createInstance('<plugin_id>')` to catch any constructor-related issues.

### 13. Print next steps

1. Replace the `placeholder-model-id` in `getConfiguredModels()` with real values, or wire up a dynamic fetch (see Anthropic's `fetchAvailableModels()`).
2. Update `definitions/api_defaults.yml` with real `max_tokens` defaults and any provider-specific configuration knobs.
3. Override `getSetupData()` to set sensible `default_models[<operation>]` entries for the post-install setup wizard.
4. Create a Key entity at `/admin/config/system/keys` for the API credential.
5. Visit `/admin/config/ai/providers/<plugin_id>` to configure (the route was generated in step 7).
6. Visit `/admin/config/ai/explorers` (the AI API Explorer) to test end-to-end.
7. For `custom` base only: implement the operation methods (`chat()`, etc.) — see the operation traits in `<webroot>/modules/contrib/ai/src/Traits/OperationType/` for boilerplate reduction.

## Pitfall checks before reporting done

- [ ] No constructor on the provider class — `AiProviderClientBase::__construct()` is `final`.
- [ ] On the `openai` base, NO operation interfaces are redeclared on `implements` (the base provides them).
- [ ] `getSupportedOperationTypes()` matches the implemented interfaces exactly.
- [ ] `definitions/api_defaults.yml` is keyed by operation type (NOT by `models`), with `input` / `authentication` / `configuration` sub-keys per op.
- [ ] No raw secrets in `config/install/*.yml`.
- [ ] Plugin path is exactly `src/Plugin/AiProvider/` (PascalCase, singular).
- [ ] `<module_name>.info.yml` includes `ai:ai` and a `configure:` line pointing to the settings form route (`key:key` is optional hardening — the bundled Anthropic provider omits it).
- [ ] `drush cr` ran without error.
